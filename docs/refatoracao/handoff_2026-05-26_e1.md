**Pontos de decisão pendentes (perguntar antes de codar)**:

1. **Clean break vs flag de transição.** Norte Seção 5 prefere clean
   break ("sem usuários = sem retrocompat"). Argumentos:
   - **A favor de clean break**: alinha com norte; relatório E.0
     pós-H-A1 mostra wins do CSP (H-R1 zerado em costas Full Body
     vs antigo 15%; overlap R-1 melhor em todas 4 configs); âncoras
     obrigatórias 0% violação; viabilidade 100/100.
   - **A favor de flag temporária**: ABC 3T agora 25s/rotina (vs
     5ms antigo) é o pior caso percebido (p95 ~30s). Se a UI sentir,
     flag `?motor=legacy` ou checkbox dá rollback rápido sem precisar
     reverter merge. Custo: 1 if/else adicional no adapter; vale a
     pena se houver dúvida sobre tempo.
   - **Recomendação implícita**: clean break (alinhamento com norte).
     Mas pergunta antes de codar.

2. **Bíceps família única** (achado #1 da E.0 não resolvido):
   `biceps(2)` no CSP é INFEASIBLE (H-T1 hard intra-treino) porque
   as 6 variações compartilham `variacao_de="Rosca bíceps"`. O
   antigo aceita via relax intra-família. Caminhos:
   - (a) **Cadastrar `Rosca martelo`, `Rosca direta` etc com
     `variacao_de` diferentes** — decisão clínica (são variações
     do mesmo movimento? "rosca pegada neutra" vale como família
     separada?).
   - (b) **Mudar H-T1 hard → soft** com peso alto. Contradiz norte
     Seção 4 ("não preservamos greedy"); H-T1 é hard há decisões
     fundadoras.
   - (c) **Aceitar como NO-OP**: ABC clássico não exige bi(2); a
     UI pode oferecer `bracos(2)` como alternativa (mistura bi+tri).
     **Já é o caminho usado pelo harness E.0** (Day B usa `bracos(2)`).
   - **Sugestão pra E.1**: decisão (c) — manter NO-OP, documentar
     na UI (futuro). Não bloqueia E.1; só vira aviso se user pedir
     `biceps(2)` direto.

3. **Formato dos avisos `h_a1_aplicadas`/`h_r1_aplicadas` na UI.**
   Quando `degraded=True`, o motor já marca motivo (`pool sem
   candidato`, `conflito_cardinalidade`). Pergunta:
   - Propagar como `Sessao.avisos` separados (novo `tipo:
     "ha1_degraded"` / `tipo: "hr1_degraded"`)?
   - Ou agregar num único `tipo: "constraint_degraded"` com payload
     genérico?
   - Modal `_avisos_modal.html` precisa de extensão? (provavelmente
     sim, mas formato fica pra decisão).

4. **`/gerar` precisa de `historico_r1=`?** O `/regerar` da Frente
   C ainda não passa histórico — passa `historico_r1_por_treino=`
   apenas quando toggle UI está ON. Pra `/gerar` (rotina nova), o
   `historico_r1` faria sentido pra evitar overlap com a rotina
   ativa anterior (UX do toggle "Usar histórico R-1"). Confirmar
   se essa parte está dentro do escopo da E.1 ou se vira refinamento
   posterior.
