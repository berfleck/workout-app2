## Achado 1 — Distribuição lower 2:1 sempre favorecendo anterior; panturrilha zerada

### Sintoma

Para `regiao lower(3)`, ambas rodadas (media e alta) deram **2
perna_anterior + 1 perna_posterior + 0 panturrilha** em CADA treino. Não há
variação cross-treino — T1 e T2 caem no mesmo split. O viés sobrevive à
mudança de aderência.

Crítica clínica: para `lower(3) × 2 treinos = 6 slots`, a distribuição
natural é 3 anterior + 3 posterior. Mesmo aceitando 2:1 dentro de um treino
(forçado por paridade), a rotina deveria alternar entre treinos (T1=2+1,
T2=1+2). Panturrilha nunca aparecer também é clinicamente errado quando o
usuário escolheu "lower" abrangente.

### Causa estrutural

- `ANCORAS_POR_REGIAO['lower']` em `gerador_treino.py:148` tem pesos 2/2/1
  (ant/post/pant). Pesos iguais entre ant/post + 3 vagas → alocação
  Hamilton determinística com tie-break fixo. Mesmo lado ganha sempre.
- Panturrilha tem `obrigatoria=False` — dropada quando vagas não cobrem
  todos os pesos. Em `lower(3)` ela some 100% do tempo.
- CSP não tem soft cross-treino "balancear distribuição de subregião
  dentro de uma região". Cada treino decide independente, ambos caem no
  mesmo viés.

### Caminho proposto

Combinação dos 3 lados:

(a) Aleatorizar tie-break do Hamilton por seed quando pesos empatam —
trivial.

(b) Eixo soft novo "balanço subregião cross-treino dentro de região"
(análogo ao S-A1 mas cross-treino) — premia distribuições simétricas
entre treinos da rotina.

(c) Repensar `obrigatoria=False` de panturrilha: ou subir para
`obrigatoria=True` (quebra rotinas curtas), ou tratar regiões com
"acessórios opcionais" via mecanismo separado (não Hamilton puro).

### Status

Aberto. Decisão clínica primeiro (qual o comportamento esperado quando
`lower(3)`), depois implementação.

### Interações com roadmap

- Item `S-R1 — distribuição multi-eixo entre treinos` (Bloco 4) cobre
  parcialmente (b). Reaproveitar.
- `S-T4 — variedade de eixos dentro de subregião com múltiplos slots`
  (Bloco 4) tangencia mas é INTRA-treino.

## Achado 2 — Equipamento repetido cross-treino para padrão single-slot

### Sintoma

Rodada com `aderencia=alta`:
- Peito: T1 = Supino Inclinado Halteres + T2 = Supino Com Halteres
  (2 supinos halteres, ambos Principal)
- Ombro: T1 = Desenv. Halteres Uni. + T2 = Desenv. Halteres Sentado
  (2 desenvolvimentos com halteres, um Principal um Intermediário)

Em ambos casos havia opções não-halteres no banco (Supino Com Barra,
Supino Smith Inclinado para peito; Desenv. Landmine, Desenv. Smith para
ombro). Variedade de equipamento foi totalmente perdida.

### Causa estrutural

- As dimensões `pegada` e `equipamento_grupo` (sobreviventes da Etapa 6
  do refator antigo — `dimensoes_proximidade.md` Seções 1-2) têm
  semântica INTRA (par-a-par dentro do mesmo escopo same-treino). Para
  padrão single-slot per treino (peito 1 + ombro 1), INTRA nunca dispara.
- O INTER cross-treino dessas dimensões vive em ~0.8 × INTRA (decisão da
  Seção 8.9/D3.1) e é integrado no `_score_proximidade` do gerador
  antigo. **No CSP novo essa lógica não foi reaproveitada** —
  pegada/equipamento_grupo cross-treino não competem no objetivo.
- Mesmo se competissem, com `peso_aderencia=2` dominando, o INTER de
  equipamento seria engolido.

### Caminho proposto

Eixo S- novo (sugestão: **S-E1 — diversidade de equipamento cross-treino
dentro do mesmo padrão**). Peso suficiente para discriminar entre 2
Principals do mesmo padrão quando equipamento difere vs quando coincide.
Cuidado: não pode forçar variedade quando o banco só tem 1 equipamento
para o padrão (degradação graceful).

### Status

Aberto. Solução estrutural — calibração não resolve.

### Interações com roadmap

- Item `S-R3 — variedade no nome dentro da rotina` (Bloco 4) é
  parcialmente relacionado mas mais raso (nome != equipamento).
- Tangencia o catálogo da `dimensoes_proximidade.md` — mas o CSP novo
  não consome aquele catálogo; teria que ser declarado em
  `catalogo_constraints.md`.

## Achado 3 — Supersets pareando exercícios da mesma região

### Sintoma

Rodada com `aderencia=alta`:
- T1 Bloco B: Passada (lower) + Stiff Uni. Smith (lower) — **2 lower**
- T1 Bloco C: Remada LM Neutra (upper) + Desenv. Halteres Uni. (upper) — **2 upper**
- T1 Bloco D: Crunch Na Bola (core) + Infra Roll-Up (core) — **2 core**
- T2 Bloco D: V-Up Unilateral (core) + Abd Bicicleta (core) — **2 core**

Metade dos blocos (4/8) parou um exercício com outro da mesma região.
Crítica clínica: superset existe para alternar grupos musculares — um
descansa enquanto o outro trabalha. Pares same-region anulam esse
benefício; viram circuito do mesmo grupo, com fadiga cumulativa errada.

### Causa estrutural

- Gerador antigo greedy tinha geo-diversidade P1-P4 em `montar_blocos`
  (CLAUDE.md menciona). Heurística ordenava blocos preferindo regiões
  diferentes nos slots ex1/ex2/ex3.
- **CSP novo não tem constraint análoga.** O único soft pareamento
  intra-bloco é `S-B1 distância funcional + evitar_agonistas` (Fatia
  4.B) que mira agonistas específicos, não "regiões diferentes em
  supersérie". Quando o usuário não liga `evitar_agonistas` (default
  OFF), nada premia diversidade de região.
- Feature **perdida na migração greedy → CSP**, não foi documentada
  como gap em nenhum log de Fatia.

### Caminho proposto

Eixo S- novo (sugestão: **S-B5 — diversidade de região INTRA-bloco**).
Soft que premia blocos com ex1.regiao ≠ ex2.regiao ≠ ex3.regiao.
Possivelmente integrado em ConfigVariedade ou ativo por padrão.
Crítica: pode tornar inviável certas demandas (tipo upper(4) sozinha
sem outras regiões) — precisa fallback graceful.

### Status

**✅ FECHADO 2026-05-27** via Frente **S-B5 diversidade de região
INTRA-bloco** (branch `sb5-diversidade-regiao-bloco`, aguarda merge
FF). Soft constraint nova no objetivo do CSP: penalty por par mesmo
bloco + mesma região (peso=4, default ON sem toggle UI). Skip
estrutural em treinos single-region (otimização de tempo).

Resultado:
- Full Body 2T (sub): 38.8% → 0% blocos same-region.
- Full Body 2T (região, este achado): 22.5% → 0%.
- Smoke E2E `/gerar` confirma 0/8 blocos same-region vs 4/8 da
  auditoria N=1.

Ver `logs/mvp_sb5_diversidade_regiao_bloco.md` para detalhes.

### Interações com roadmap

- Não tem item específico no roadmap atual.
- Tangencia `S-B2 — balanço de carga implícita` e `S-B3 — fadiga
  prévia no bloco` (Bloco 4) mas é semanticamente distinto.

## Achado 4 — Calibração default de `peso_aderencia` mata o sistema de tier

### Sintoma

Rodada com `aderencia=media` (default da UI atual): **0 de 16 slots**
caíram em exercício tier Principal. Peito caiu em Acessório (Crucifixo
Halteres) + Intermediário (Apoio Ajoelhado); perna_anterior caiu em
Intermediário (Goblet Rampa) + Acessório (Cadeira Extensora).

Rodada com `aderencia=alta`: **11 de 16 slots** Principal. Diferença
brutal vinda de mudar um único peso de 0 → 2.

### Causa estrutural

`app_flask.py:486`:

```python
_PESO_ADERENCIA_POR_PERFIL = {"alta": 2, "media": 0, "baixa": 0}
```

Dois problemas:

1. **`media == baixa == 0`** — perfis distintos tratados iguais. Não há
   forma de modelar "aluno com baixa aderência" diferente de "médio".
2. **`media = 0` significa "ignora tier completamente"** — sorteio
   uniforme do pool. Vai contra prática clínica padrão de prescrever
   Principal por default.

Comentário no próprio código admite: *"Chute inicial conservador.
Iterar pra cima se calibração indicar."* Calibração nunca foi feita.
Frente D foi tunada em **um** smoke (hinge(1)) sem matriz mais ampla.

### Caminho proposto

Calibração curada de partida (ajuste depois com dashboard quantitativo
do Bloco 5):

- `alta = 3-4` — forte preferência por Principal, pouca variedade entre
  tiers
- `media = 1-2` — Principal preferido, variedade dentro de tier
  preservada
- `baixa = 0` ou negativo — favorece Intermediário/Acessório, espelha
  preparação ou destreinamento

Subir só `media` para 1 já cobriria 80% do dia-a-dia, sem precisar
mudar `alta`.

### Status

**🔴 Aberto, bloqueia uso real.** Personal trainer não vai trocar
aderência a cada aluno só para conseguir Principal nos slots. Default
errado vira viés sistemático em todas as rotinas geradas pela UI.

### Interações com roadmap

- Item `Gate de avaliação clínica semântica` (Bloco 4) tem essa
  calibração como dependência implícita.
- Item `Dashboard quantitativo` (Bloco 5) é o instrumento formal de
  calibração — mas calibração de partida não precisa esperar dashboard,
  pode sair do PT direto (curated weight).
- Frente D (concluída) entregou a infraestrutura — calibração é o que
  falta.
